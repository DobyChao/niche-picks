> 最后更新：2026-05-18
> 
> 状态：Ready for Implementation

## 1. 产品定位

私人 / 小圈子的探店日记应用。记录真实的探店体验，数据本地优先，一键按需同步，管理员整批审核。

**核心价值**：不依赖商业点评，小圈子内“自己记、朋友看、管理员审”。

## 2. 数据模型 (本地与远端通用)

为了确保离线删除操作能正确同步到其他客户端，所有核心实体均采用**墓碑机制（软删除）**。

### 2.3 变更日志项 (ChangeLogItem)

客户端本地记录未提交的变更，用于 push 时的 payload。

| **字段**      | **类型**                             | **说明**                                       |
| ----------- | ---------------------------------- | -------------------------------------------- |
| `entity`    | `"shop" \| "review"`               | 变更实体类型                                       |
| `entityId`  | `string`                           | 对应数据的唯一 ID                                   |
| `action`    | `"create" \| "update" \| "delete"` | 操作类型（delete 时 snapshot 中 `isDeleted = true`） |

| `timestamp` | `string`                           | 操作产生的 ISO 8601 时间戳                           |

## 3. 核心业务流 (同步与一键审批)

本项目的核心是**一键整批审批流**，它抛弃了逐字段 Diff Cherry-pick 的复杂性。

### 3.1 变更提交流程 (Push)

1. **本地记录**：用户在本地 IndexedDB 进行任何增删改操作，均静默生成对应的 `ChangeLogItem` 写入本地变更历史。

2. **发起推送**：用户点击“同步数据”，输入其个人 User Token。

3. **数据封装**：客户端打包所有本地尚未 push 的 `ChangeLogItem`，附带昵称和时间戳。

4. **服务端挂起**：服务端接收到该 batch 后，作为一条 **Pending 记录** 写入数据库，状态设为 `pending`。

5. **本地清理**：客户端收到服务端接收成功的响应（返回 `syncId`）后，**不清除**本地实体数据，但**清空**本地的 `ChangeLog`，防止重复提交。

### 3.2 一键审批流程 (Approve / Reject)

1. **待办列表**：管理员登录审核页面，拉取所有的 Pending 列表。

2. **列表卡片展示**：每个 Pending 单元卡片展示：
   
   - 提交人昵称、提交时间。
   
   - 变更摘要（例如：“新增 2 个店铺，修改 1 条点评，删除了 1 个店铺”）。

3. **一键通过 (Approve)**：
   
   - 策略：**整批覆盖合并 (LWW - Last Write Wins)**。
   
   - 操作：服务端将该批次中包含的所有最新数据快照（Snapshot）**直接全量覆盖写入**服务端的 Shop 与 Review 主数据表。
   
   - 该批次状态更新为 `approved`。

4. **一键拒绝 (Reject)**：
   
   - 操作：服务端不将该批次的数据合入主表，直接标记该批次为 `rejected`。

### 3.3 数据拉取与回滚 (Pull & Sync Status)

1. **客户端拉取**：用户执行 Pull 操作，传入 `lastSyncTime`。

2. **服务端响应**：
   
   - 返回自 `lastSyncTime` 之后，主表中所有有更新（包括 `isDeleted=true`）的数据。
   
   - 同时返回该用户此前提交的 Pending 批次的最终处理结果（`approved` / `rejected`）。

3. **客户端合并**：
   
   - 收到最新的主数据：本地直接使用 `id` 进行覆盖（若主数据中某条记录 `isDeleted` 为 true，本地也同步设为真）。
   
   - 收到 `rejected` 结果：说明本地之前的某些离线修改未通过审核。客户端会将对应实体恢复（Rollback）至本次拉取到的云端标准版本。

4. **状态标志位**：数据状态在客户端通过一个计算属性呈现给用户：
   
   - `local-only` (本地未提交)
   
   - `pending` (已提交等审核)
   
   - `synced` (已同步一致)

## 4. 交互与响应式规则

### 4.1 数据来源可视化表现

为了保留极客掌控感，每一张店铺/点评卡片和地图 Marker 均有直观的来源标识：

- 🟢 **本地未提交**：仅本地编辑了，尚未 push（或 push 处于 pending 状态）。

- 🔵 **已同步**：数据已合入远端主线，且与本地版本一致。

### 4.2 响应式断点 (768px)

- **移动端 (< 768px)**：
  
  - 地图常驻在屏幕上半部分 (占比 40vh)。
  
  - 搜索框与列表堆叠在下半部分 (可向上滑动拉起呈大图层，防止地图遮挡)。

- **桌面端 (≥ 768px)**：
  
  - 左侧大面积渲染地图（占宽度的 `calc(100% - 420px)`）。
  
  - 右侧固定为宽度 `420px` 的多功能控制面板（承载搜索框、分类标签、列表以及详情内容），确保内容在宽屏下不拉伸、不易读。

## 5. Token 权限规范

- **User Token** (`usr_<random32>`)：仅具有拉取（pull）和推送（push）自身变更的权限。

- **Admin Token** (`adm_<random32>`)：具有审批通过/拒绝、拉取 pending 列表的完整读写权限。

## 6. MVP 范围限制

### 🟩 MVP 包含

- 店铺 + 点评 本地完整 CRUD。

- 软删除（墓碑机制）同步。

- 本地 IndexedDB 事务级写入 + 服务端 SQLite 高并发数据合并。

- 极简的一键“整批通过”/“一键拒绝”审批台。

- 响应式地图布局（含 GCJ-02 纠偏声明）。

### 🟥 MVP 排除

- 图片存储与展示（V2 解决）。

- 对比差分（Diff）明细视图（不展示单个字段的修改差异，仅展示实体级别变动）。

- 撤销审核、自动静默同步。
